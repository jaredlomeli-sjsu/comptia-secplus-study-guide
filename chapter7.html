Denial of Service

Overwhelm a resource until it can't serve anyone

A Denial of Service (DoS) attack makes a system or service unavailable to legitimate users — almost always through resource exhaustion, consuming bandwidth, memory, CPU, or connection tables until nothing is left for real requests. A Distributed DoS (DDoS) uses many machines at once — typically a botnet of compromised hosts — so the flood is far larger and can't be stopped by blocking one IP.

AttackHow It WorksIdentify It By
DoSA single source exhausts a target's resourcesOne attacking host overwhelms a service
DDoSMany sources (a botnet) flood the target at once"Traffic from thousands of IPs worldwide"
SYN floodSends a storm of TCP SYN packets but never completes the handshake, filling the connection table with half-open connectionsResource exhaustion of TCP state; mitigated by SYN cookies
Reflected DDoSAttacker spoofs the victim's IP and sends requests to third-party servers, which reply to the victimVictim is buried by responses it never asked for
Amplified DDoSA reflected attack using protocols where a tiny request triggers a huge reply (DNS, NTP, memcached)Small spoofed query → massive response multiplies the flood
Indicator vs attack: an indicator is the observable sign of an attack — a traffic spike, a flood of half-open connections, a service timing out. The exam often gives you the indicator and asks you to name the attack behind it.

Spoofing & On-Path Attacks

Faking identity and sitting in the middle of a conversation

Spoofing is falsifying an identifier — a source IP, MAC, email address, or caller ID — to impersonate a trusted source. Forgery is the broader act of creating fraudulent data (a forged request, token, or certificate) so a system treats it as genuine. Both set up the bigger attacks below.

AttackWhat HappensDefense
On-path attack (MITM)Attacker secretly relays or alters traffic between two parties who think they're talking directly — the modern name for man-in-the-middleEnd-to-end encryption (TLS), mutual authentication
Attacker-in-the-browser a.k.a. man-in-the-browserMalware/extension inside the browser alters pages and transactions after TLS decrypts them — encryption can't helpEndpoint anti-malware, extension control, transaction verification
SSL strippingAn on-path attacker downgrades a victim's HTTPS connection to plain HTTP, then reads everything in cleartextHSTS, redirect-to-HTTPS, certificate checks
Replay attackCaptures valid traffic and re-sends it to repeat an action or impersonate a userNonces, timestamps, session tokens
Credential replayA replay attack that reuses captured authentication data (a hash, ticket, or token) to log in without knowing the passwordOne-time tokens, MFA, short-lived tickets
Encryption isn't always the answer. Against an on-path attack, TLS works. Against attacker-in-the-browser, it doesn't — the malware operates after decryption, on the endpoint. Against SSL stripping, the fix is forcing HTTPS (HSTS) so there's no plaintext to strip.

DNS Attacks

Poison the lookup, steal the destination

DNS turns names into IP addresses, and a URL (Uniform Resource Locator) is the full web address a user types. Because users trust the name and rarely check the resolved IP, DNS is a prime target for sending victims somewhere malicious.

AttackWhat Happens
DNS poisoning (cache poisoning)Corrupting a DNS resolver's cache so a legitimate name resolves to the attacker's IP
PharmingThe result of poisoning at scale: silently redirecting users from a legitimate site to a fake one — no click required
URL redirectionSending a user to a different destination than expected — via a malicious link, a compromised page, or an open redirect on a trusted site
Domain hijackingTaking control of a domain's registration (stolen registrar credentials, social engineering) to reroute all of its traffic
Phishing vs pharming: phishing lures you to click a bad link; pharming redirects you automatically through poisoned DNS even when you type the correct address. Pharming needs no user mistake.

DNS & URL Defenses

Authenticate lookups, filter destinations, keep the logs

DNSSEC DNS Security Extensions

Adds digital signatures to DNS records so resolvers can verify a response is authentic and unaltered. It defeats poisoning by proving origin and integrity — note it authenticates, it does not encrypt.

DNS filtering

A DNS filter blocks resolution of known-malicious or policy-violating domains — users simply can't reach them. A common first line of web/content filtering.

DNS sinkhole

Returns a false (controlled) answer for malicious domains — pointing malware's callbacks to a dead end or a monitoring host instead of the real C2 server.

DNS log files

Records every query a network makes. Invaluable for detection: beaconing to suspicious domains, data exfiltration over DNS, and tracing which host is infected.

Secure Coding & Input Handling

Most app attacks start with untrusted input
PracticeWhat It IsWhy It Matters
Input validationChecking that input matches expected type, length, format, and range before useThe single most important defense against injection attacks
Client-side validationValidation in the browser (fast feedback for users)Convenience only — trivially bypassed, never trust it for security
Server-side validationValidation on the server, where the attacker can't tamper with itThe authoritative check — always validate server-side
HTML escaping / encodingConverting special characters (< > & ") into safe entities so input renders as text, not codeStops cross-site scripting (XSS) at output
Error handlingCatching errors and returning generic messagesVerbose errors leak stack traces, paths, and DB structure to attackers
Code obfuscationDeliberately making code hard to read/reverse-engineerRaises the bar for attackers; not a substitute for real controls

Race condition (TOCTOU)

A flaw where the outcome depends on timing between events. The classic form is Time-of-Check to Time-of-Use (TOCTOU): a resource is validated, then changed in the gap before it's used. Fixed with locking and atomic operations.

Compiler & outsourced code

A compiler translates source code into executable machine code. Outsourced code vulnerabilities arise when third-party, contracted, or library code you didn't write introduces flaws or backdoors — review and monitor it like your own.

Client-side is never security: if a question pits client-side against server-side validation, the secure answer is server-side (or "both, but enforce on the server"). An attacker controls the browser.

Web Application Security

Headers, cookies, signing, and the OWASP backbone

HTTP headers

Metadata sent with every request/response. General headers apply to both, request headers carry client info, and entity headers describe the body (content type, length). Security headers (HSTS, CSP) harden the app.

Cookies & secure cookies

A cookie stores session state in the browser. A secure cookie sets the Secure flag (sent only over HTTPS) and usually HttpOnly (hidden from JavaScript) to resist theft via XSS.

Code signing

Digitally signing software with the publisher's private key so users can verify authenticity and integrity — the code came from the stated author and wasn't tampered with.

OWASP — Open Worldwide Application Security Project

A nonprofit community behind the well-known OWASP Top 10 list of the most critical web application risks (injection, broken access control, XSS, and more). The go-to reference for secure web development and testing.

Injection & Memory Attacks

When untrusted input becomes code or corrupts memory

A database stores data in records and fields (columns), queried with SQL (Structured Query Language). When an app builds a SQL query by gluing user input directly into the statement, an attacker can inject their own SQL — a SQLi (SQL injection) attack — to read, alter, or destroy data.

AttackWhat It TargetsDefense
SQL injection (SQLi)Injects SQL through unsanitized input to manipulate the databaseParameterized queries / stored procedures + input validation
LDAP injectionInjects into LDAP directory queries to bypass auth or read the directoryInput validation, escaping LDAP metacharacters
XML injectionInjects malicious content into XML input/documents (incl. XXE)Validate/disable external entities, sanitize input
Directory traversalUses ../ sequences to escape the web root and read arbitrary filesCanonicalize paths, validate file names, least privilege
Reflected XSSMalicious script in a request is reflected back and runs in the victim's browser (via a crafted link)Output encoding, input validation, CSP
Stored XSSScript is saved on the server (a comment, profile) and runs for every visitorOutput encoding, input validation, CSP
DLL injectionForces a process to load a malicious Dynamic Link Library to run code in its contextCode signing, app allow-listing, EDR

Parameterized queries

A stored procedure is reusable SQL stored in the database. A parameterized stored procedure binds user input as data, never as executable SQL — the definitive fix for SQLi.

Memory flaws

Buffer overflow: writing past a buffer's bounds to corrupt memory/hijack execution. Integer overflow: a value exceeds its type's max and wraps around. Memory leak: failing to free memory until the app exhausts it.

Web server logs

Record every request to the server. After an injection or traversal attempt, the logs show the malicious payloads, source IPs, and which URLs were targeted — key evidence for detection and IR.

Automation & SOAR

Scaling security operations with scripts and playbooks

SOAR (Security Orchestration, Automation, and Response) connects your security tools and runs playbooks — automated, repeatable response steps — so routine actions happen instantly and consistently without an analyst doing them by hand.

Use caseWhat Gets Automated
User provisioningCreating, modifying, and disabling user accounts as people join, move, or leave
Resource provisioningSpinning up servers, storage, and services on demand (often via Infrastructure as Code)
GuardrailsAutomated policy limits that prevent unsafe actions (e.g., block creating a public storage bucket)
Security groupsProgrammatically managing firewall/access rules around resources
Ticket creationAuto-generating incident/help-desk tickets when an alert fires
EscalationRouting an unhandled or high-severity issue to the right team automatically
Enabling / disabling services & accessTurning accounts, ports, or services on or off in response to events
Continuous integration / testing (CI/CD)Automatically building, testing, and deploying code on every change
Integrations & APIsThe APIs that let tools talk to each other — the glue that makes orchestration possible

Automation & Scripting Trade-offs

The benefits the exam loves — and the real costs

Benefits

  • Efficiency / time savings — machines do repetitive work instantly
  • Enforcing baselines — every system configured the same way
  • Standard infrastructure configurations — repeatable, predictable builds
  • Secure scaling — grow without growing mistakes
  • Employee retention — less tedious manual toil
  • Faster reaction to incidents — automated response in seconds
  • Workforce multiplier — a small team does the work of many

Drawbacks

  • Increased initial costs — tooling and setup investment up front
  • Complexity — automated systems are harder to understand
  • Single point of failure — one broken script can break everything
  • Technical debt — quick scripts accumulate and rot over time
  • Ongoing supportability — automation must be maintained as systems change
Exam framing: automation's headline wins are efficiency, consistency (baselines), and acting as a force multiplier. Its headline risks are complexity, single points of failure, and technical debt. Match the scenario to the right column.
