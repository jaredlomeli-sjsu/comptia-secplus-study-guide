Hashing & Integrity

One-way transformation — same input always produces the same fixed-length output

A hash (also called a message digest) is the result of running data through a hash function — a one-way mathematical algorithm that converts any amount of input into a fixed-length string. Hash functions are deterministic (same input → same output) but irreversible. They are the backbone of data integrity, digital signatures, and password storage.

Hash Algorithm Families

SHA-3224 / 256 / 384 / 512-bitCurrentStandardized by NIST in 2015. Uses a completely different internal structure (Keccak sponge) from SHA-1/SHA-2. Designed as a backup standard if SHA-2 were broken. Resistant to length-extension attacks.

HMAC — Hash-Based Message Authentication Code

A Hash-Based Message Authentication Code (HMAC) combines a cryptographic hash function with a secret key. Unlike a plain hash (integrity only), an HMAC also proves authenticity — that the message came from someone who knows the shared key.

Plain Hash

Verifies integrity only. Anyone who receives the data can compute and compare the hash. No secret needed. Does not prove who sent the message.

Use when: File download verification, blockchain, password storage.

HMAC

Verifies integrity + authenticity. Requires a shared secret key. Only parties with the key can generate or verify valid HMACs. Prevents forgery even if traffic is intercepted.

Use when: API authentication, TLS record-layer MAC, IPsec.

Non-Repudiation & Checksum

Non-Repudiation

The inability to deny having performed an action. A digital signature achieves non-repudiation — only the holder of a private key can produce that signature, so they cannot later deny signing. A plain HMAC does not provide non-repudiation because both parties share the key.

Checksum

A simple error-detection value (e.g., CRC32) that detects accidental corruption but is not collision-resistant and provides no security against intentional tampering. Use a cryptographic hash when security matters.

Hash Collision

A hash collision occurs when two different inputs produce the same hash output. Because hash functions map infinite inputs to a finite output space, collisions must theoretically exist — the goal is to make them computationally infeasible to find intentionally. MD5 and SHA-1 have known practical collision attacks; SHA-256 and SHA-3 do not.

Collision danger: An attacker who engineers a hash collision can substitute a malicious document for a legitimate one while producing the same hash — bypassing integrity checks entirely. This is why MD5 and SHA-1 are unsuitable for digital certificates and code signing.
Exam tip: SHA-256 is part of the SHA-2 family and is the current standard. SHA-1 is deprecated (SHAttered collision, 2017). MD5 is broken for security. SHA-3 uses a completely different internal structure (Keccak sponge). HMAC = hash + shared secret key.

Password Attacks & Defenses

Online vs. offline · cracking techniques · salting & key stretching

Password attacks split into online attacks (against a live authentication system, limited by lockout) and offline attacks (against a stolen hash database, running at billions of attempts per second on GPU hardware with no lockout).

AttackModeHow It WorksPrimary Defense
Online attackOnlineSubmitting credentials to a live login interface (web form, SSH, RDP). Limited by account lockout, rate limiting, and IDS/IPS detection.Account lockout, MFA, rate limiting, CAPTCHA.
Offline attackOfflineAttacker obtains a stolen hash database and cracks hashes locally. No lockout applies — speed is limited only by hardware.bcrypt / Argon2, salting, key stretching.
Dictionary attackBothTries passwords from a wordlist of common passwords, phrases, and previously leaked credentials. Effective against weak or predictable passwords.Block known-breached passwords, enforce complexity.
Brute force attackBothSystematically tries every combination of characters up to a length. Guaranteed to succeed eventually — impractical against long passwords.Long passwords (16+ chars), account lockout, MFA.
Spraying attackOnlineTries a small set of common passwords (e.g., Summer2024!) against a large number of accounts. Stays below per-account lockout thresholds by spreading attempts.Monitor for distributed failures across many accounts; ban common passwords.
Pass the hashOnlineAttacker steals the NTLM credential hash from memory (via Mimikatz) and replays it directly — never needing the plaintext. The hash is the credential.Credential Guard, Protected Users group, disable NTLM, Privileged Access Workstations.
Birthday attackOfflineExploits the birthday paradox to find any two inputs producing the same hash. Reduces collision work from 2n to 2n/2.Use hash algorithms with large digest sizes (SHA-256+).
Rainbow table attackOfflinePrecomputed table of hash values for millions of common passwords. Lookup is near-instant — find the hash, read the plaintext. Much faster than brute force against unsalted hashes.Salting — a unique random salt per password makes precomputed tables useless.
Plaintext / known-plaintextOfflineAttacker has both plaintext and its encrypted/hashed form and uses the pair to analyze the algorithm or derive keys. Targets weak ciphers or predictable key generation.Strong modern algorithms; random initialization vectors (IVs).
Pass the hash: Resetting the user's password does not help — the attacker cached the old hash and will keep replaying it until the session is invalidated. Credential Guard (Windows) prevents hash extraction from LSASS memory.

Password Defenses — Salting & Key Stretching

Salting a Password

A salt is a random unique value appended to a password before hashing. Each account gets a different salt, so two users with the same password produce completely different stored hashes.

Defeats: Rainbow table attacks (precomputed tables become useless) and hash comparison (identical passwords no longer match in the database).

The salt is stored alongside the hash — it is not secret, but it must be unique per user.

Key Stretching

Deliberately makes the hash function slow by repeating it thousands or millions of times. A fraction-of-a-second delay is acceptable for legitimate logins but multiplies attacker cost by the same factor.

GPU rigs that compute billions of plain SHA-256 hashes per second are drastically slowed by bcrypt or Argon2, which are designed to resist parallelization.

Key Stretching Algorithms

AlgorithmBased OnKey FeatureCommon Use

PBKDF2Iterative HMAC (typically HMAC-SHA-256)Password-Based Key Derivation Function 2. Configurable iteration count with a salt. FIPS 140-2 compliant. Widely supported across platforms.WPA2/WPA3 Wi-Fi passwords, iOS device encryption, PKCS#5.

Exam tip: Salting defeats rainbow tables. Key stretching defeats GPU brute-force speed. bcrypt, PBKDF2, and Argon2 incorporate both. Plain SHA-256 — even with a salt — is not suitable for password storage because it is too fast.

Symmetric Encryption

Same key encrypts and decrypts — fast bulk encryption with a key distribution challenge

In symmetric encryption, the same secret key both encrypts and decrypts data. Both parties must share an identical key before communication. Symmetric algorithms are far faster than asymmetric ones and are used for bulk data (files, disk, network traffic). The fundamental challenge is key distribution — how do two parties share a key securely without it being intercepted?

Algorithm vs. Key

Every cipher has two components: the algorithm (the mathematical procedure) and the key (the secret value parameterizing it). The same algorithm with a different key produces completely different ciphertext. A larger key space means exponentially more work to brute-force — doubling key length squares the effort required.

Symmetric Algorithm Comparison

AlgorithmKey SizeBlock SizeStatusNotes
AES128 / 192 / 256-bit128-bitCurrent standardAdvanced Encryption Standard, selected by NIST in 2001. AES-256 is used by the US government for TOP SECRET data. No practical attacks. Used in TLS, disk encryption (BitLocker, FileVault), WPA3, and virtually every modern system requiring symmetric encryption.
3DES112 or 168-bit effective64-bitDeprecatedTriple DES. Applies DES three times (Encrypt–Decrypt–Encrypt). DES was broken in 1997; 3DES extended its life. NIST deprecated 3DES in 2023 — the 64-bit block size enables SWEET32 birthday attacks on long TLS sessions. Being replaced by AES everywhere.
Blowfish32–448-bit (variable)64-bitLegacyDesigned by Bruce Schneier in 1993 as a fast, unpatented alternative to DES. Variable key length is flexible but the 64-bit block shares the SWEET32 weakness with 3DES at high data volumes. Largely superseded by its successor Twofish and by AES.
Twofish128 / 192 / 256-bit128-bitSecureSchneier's successor to Blowfish. AES finalist (lost to Rijndael). 128-bit block eliminates birthday-attack concerns. No practical attacks known. Available in many libraries; less common than AES in practice.

Block Ciphers vs. Stream Ciphers

Block Cipher

Encrypts data in fixed-size blocks (AES: 128-bit blocks). The whole block must be available before encryption starts. Requires a mode of operation (CBC, GCM, CTR) to handle messages longer than one block and provide authentication where needed.

Algorithms: AES, 3DES, Blowfish, Twofish.

Best for: File encryption, disk encryption, stored data — any situation where the full data is available before processing.

Stream Cipher

Encrypts data one bit or byte at a time as it flows. Generates a keystream (pseudorandom sequence) and XORs it with plaintext. No need to wait for a full block — ideal for real-time data.

Algorithms: RC4 (deprecated), ChaCha20 (modern — used in TLS 1.3 on mobile).

Best for: Network streams, voice/video, low-latency applications where data size is unknown.

Exam tip: AES is the answer for symmetric encryption on any modern system. 3DES is legacy/deprecated. The 64-bit block of 3DES and Blowfish creates SWEET32 vulnerability at high volumes. Key size and block size are different — AES-128 uses a 128-bit key and a 128-bit block, but those are coincidentally equal.

Asymmetric Encryption & Digital Signatures

Key pairs solve key distribution — public key in, private key out

In asymmetric encryption, mathematically linked key pairs are used: a public key (freely shared) and a private key (kept secret by the owner). Data encrypted with the public key can only be decrypted with the private key — and the reverse holds for digital signatures. Asymmetric algorithms are slower than symmetric ones; in practice they encrypt a symmetric session key rather than bulk data (hybrid cryptography).

Key Exchange

Key exchange is the process of securely establishing a shared secret over an untrusted network. In hybrid cryptography (TLS), asymmetric cryptography exchanges a symmetric session key; all bulk data is then encrypted with the faster symmetric key.

Digital Signature Algorithms

Ephemeral Keys, Static Keys & Perfect Forward Secrecy

Static Key

A long-lived key reused across many sessions. If ever compromised, all past and future sessions encrypted with it can be decrypted. Simpler to manage but one breach exposes all historical traffic.

Ephemeral Key

A temporary key generated fresh for each session and discarded immediately after use. Even if an attacker records traffic and later compromises a long-term private key, they cannot decrypt past sessions — each session used a unique throwaway key.

Perfect Forward Secrecy (PFS)

Achieved by using ephemeral Diffie-Hellman (DHE or ECDHE) key exchange so that compromise of long-term keys cannot decrypt past sessions. TLS 1.3 mandates PFS. Look for DHE or ECDHE in cipher suite names.

Exam tip: PFS = ephemeral keys (DHE/ECDHE). Private key signs / decrypts. Public key verifies / encrypts. ECC = smaller keys, same security. ECDSA = ECC-based digital signature algorithm. TLS 1.3 requires PFS and forward-secrecy cipher suites.

Obfuscation, Steganography & Tokenization

Obfuscation

Making data or code difficult to understand without necessarily encrypting it — code minification, variable renaming, data masking. Provides security through obscurity. Not strong security on its own; combine with encryption for sensitive data.

Steganography

Hiding data inside other data without visibly altering the carrier. The goal is concealment, not just confidentiality.

  • Image — data hidden in least-significant pixel bits.
  • Audio — data embedded in inaudible frequencies or low-order bits.
  • Video — data hidden in individual frames.

Tokenization

Replacing sensitive data (e.g., credit card number) with a non-sensitive token that maps back to the real value only inside a secure vault. The token has no mathematical relationship to the original — it cannot be reversed without the vault. PCI-DSS standard practice.

Steganography in attacks: Malware and data-exfiltration tools embed C2 traffic or stolen data inside innocuous image/audio files to evade DLP tools that scan for recognizable sensitive patterns.

Digital Certificates

X.509 structure — binding a public key to an identity, signed by a trusted CA

A digital certificate is an electronic document that binds a public key to an identity (a server, person, device, or organization). Certificates follow the X.509 standard and are digitally signed by a Certificate Authority (CA). When your browser connects to an HTTPS site, it validates the site's certificate to confirm it is talking to the real server, not an impersonator.

Certificate File Types

.CER / .DER

Binary-encoded certificate. DER (Distinguished Encoding Rules) is the binary ASN.1 encoding. CER and DER are functionally the same format — a single certificate in raw binary bytes. Cannot be opened in a text editor. Common on Windows.

.PEM

Base64-encoded text format (Privacy Enhanced Mail). Recognizable by -----BEGIN CERTIFICATE----- header. Can hold a certificate, private key, or full chain. Unix/Linux standard. Opened in any text editor. OpenSSL default format.

.P7B / PKCS#7

Contains one or more certificates (no private key). Base64 text with -----BEGIN PKCS7----- header. Used to distribute CA certificate chains. Standard import format for Windows and Java certificate stores.

.P12 / .PFX

PKCS#12 archive — contains the certificate and the private key, password-protected. PFX is Microsoft's name for the same format. Used to move certificate+key pairs between systems. Always protect with a strong password.

OpenSSL

The most widely used open-source toolkit for PKI operations. Key commands: openssl req (create CSR), openssl x509 (inspect/sign), openssl pkcs12 (convert formats), openssl s_client (test TLS). Default output is PEM.

P12 / PFX contains the private key. Exporting without a strong password means anyone who obtains the file has the private key. Compromise requires immediate certificate revocation.

PKI Infrastructure

Trust chains · CA hierarchy · revocation · key management

Public Key Infrastructure (PKI) is the full framework — policies, procedures, hardware, software, and people — needed to create, manage, distribute, use, store, and revoke digital certificates. PKI lets entities that have never met establish trust by relying on a shared trusted third party: the Certificate Authority.

PKI Components

ComponentRole
Certificate Authority (CA)The trusted entity that issues and signs digital certificates. Verifies requester identity and binds their public key to that identity. Public CAs (DigiCert, Let's Encrypt) are trusted by browsers; private CAs serve internal use.
Root CATop-level CA in the hierarchy — self-signed (signs its own certificate). The root of trust. Should be kept offline and used only to sign intermediate CA certificates to minimize exposure. If the root private key is compromised, the entire PKI must be rebuilt.
Intermediate CAA subordinate CA whose certificate is signed by the Root CA. Issues end-entity certificates in day-to-day operations. Multiple intermediates create a certificate chain. Compromise of an intermediate requires only revoking that intermediate — the root is unaffected.
Registration Authority (RA)Verifies certificate requests on behalf of the CA. The RA validates requester identity and approves or denies the request; the CA does the actual signing. Separates identity verification from signing.
Certificate Signing Request (CSR)A message from an entity to a CA requesting a certificate. Contains the public key, Distinguished Name attributes, and is signed with the requester's private key to prove possession. The CA verifies and returns a signed certificate.
Certificate StoreA repository (OS or application) holding trusted certificates. Operating systems maintain a trusted root CA store — certs chaining to those roots are automatically trusted. Windows Certificate Store; browsers like Firefox use their own stores.
Root of TrustThe foundational trusted anchor — typically the Root CA certificate. Also refers to hardware trust anchors like the TPM (Trusted Platform Module) that provide a trusted starting point for device attestation and secure boot.
Certificate ChainingValidation by following the chain: end-entity certificate → intermediate CA → root CA. Each certificate is verified using the issuer's public key. The chain must terminate at a root in the trusted store.

Online vs. Offline CA

Offline Root CA

The Root CA should be kept offline — powered down, air-gapped, and stored in a physically secure location (HSM or vault). Brought online only to sign intermediate CA certificates (rarely — perhaps once a year or less).

Why: If the Root CA private key is ever compromised via a network attack, the entire PKI hierarchy must be rebuilt. No network connection = no network attack surface.

Online Issuing CA

The Intermediate/Issuing CA stays online for day-to-day operations — issuing, renewing, and revoking end-entity certificates. Automated issuance (Let's Encrypt ACME protocol) requires persistent online availability.

Risk mitigation: If the intermediate is compromised, only that intermediate is revoked and reissued — the Root CA is unaffected.

Certificate Revocation

MechanismHow It WorksTrade-offs
CRLCertificate Revocation List. The CA publishes a signed list of revoked serial numbers at a CRL Distribution Point (CDP) URL embedded in the certificate. Clients download and cache the list periodically.Lists can be large. Not real-time — a cert revoked after the last CRL download is still trusted until the next refresh (typically 24h–7d).
OCSPOnline Certificate Status Protocol. Client sends the cert's serial number to an OCSP Responder; receives a signed real-time response: Good, Revoked, or Unknown.More current than CRL. Advanced PKI Concepts

Certificate Pinning

An application hardcodes the expected certificate or public key rather than trusting any cert signed by a browser-trusted root. If the server presents a different cert (even a valid CA-signed one), the connection is rejected.

Protects against: Rogue CA compromise, fraudulent MITM certificates. Used in high-security mobile apps and APIs.

Risk: If the pinned cert expires and the app is not updated, connections break until the app is patched.

Key Escrow

Storing a copy of a private key with a trusted third party (the escrow agent). Allows data recovery if the key owner loses the key or leaves the organization.

Risk: The escrow store becomes an extremely high-value target. Law enforcement has historically pushed for mandatory key escrow to enable lawful access — strongly opposed by the security community as it weakens security for everyone.

Key Management System (KMS)

A centralized system for creating, storing, distributing, rotating, and revoking cryptographic keys. Cloud KMS products (AWS KMS, Azure Key Vault, Google Cloud KMS) manage encryption keys for cloud workloads without exposing raw key material to applications. A Hardware Security Module (HSM) is the tamper-resistant hardware that stores keys — keys never leave the HSM in plaintext.

Entropy

The randomness available for cryptographic key generation. Security depends on keys being generated from a high-quality random source. Insufficient entropy produces predictable keys. Operating systems maintain entropy pools (/dev/urandom). HSMs include dedicated hardware True Random Number Generators (TRNGs) for highest-quality entropy.

Exam tip: Root CA = offline, self-signed, root of trust. Intermediate CA = online, signs end-entity certs. CSR goes from requester to CA. CRL = periodically downloaded list; OCSP = real-time query; OCSP stapling = server pre-fetches and attaches the OCSP response. Certificate pinning rejects unexpected certs even from valid CAs.

Encoding Formats & ASN.1

Cryptographic objects (certificates, keys, CSRs) are encoded using formats defined by ASN.1 (Abstract Syntax Notation One) — an ISO/ITU standard for describing data structures independent of implementation. Multiple encoding rules exist for serializing ASN.1 to bytes.

FormatFull NameTypeNotes

ASN.1Abstract Syntax Notation OneSchema languageISO/ITU standard for describing data structures. Certificates, private keys, and CRLs are defined in ASN.1. Not itself an encoding — requires a serialization rule (BER, DER, CER).
