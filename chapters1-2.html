Chapter 1 — Mastering Security Basics

Domain 1 foundation

The CIA Triad

Every exam question maps back to one of these

Confidentiality

Only authorized parties can view data.

Enforced by: encryption, access controls, steganography.

Violated by: data breach, sniffing, shoulder surfing.

Integrity

Data hasn't been modified or corrupted.

Enforced by: hashing (SHA-256), digital signatures, version control.

Violated by: unauthorized change, MITM tampering.

Availability

Systems and data are up when needed.

Enforced by: redundancy, backups, patching, HVAC, generators.

Violated by: DoS/DDoS, ransomware, hardware failure.

Exam cue: "Verify the file wasn't altered" = integrity → hashing. "Ensure only HR can read it" = confidentiality → encryption/ACL. "Keep the site online during attack" = availability → redundancy. Match the verb to the leg of the triad.

Control guidance largely comes from NIST SP 800-series documents (e.g., SP 800-53 catalogs security and privacy controls for federal systems).

Key Stretching & Perfect Forward Secrecy

Two crypto concepts Ch 1 plants early

Key Stretching

Runs a password through thousands of hash iterations + salt so brute-force attacks become computationally expensive.

Algorithms to know: bcrypt, PBKDF2, Argon2.

A salt (random value added before hashing) defeats rainbow-table attacks.

Perfect Forward Secrecy (PFS)

Each session uses a unique ephemeral key (Diffie-Hellman Ephemeral — DHE/ECDHE). Compromising the server's long-term private key cannot decrypt past recorded sessions.

Required by TLS 1.3 — all its cipher suites are ephemeral.

Logging & SIEM

Where evidence lives
Log SourceWhat It Records
Security logAuditable events: logon success/failure, privilege use, object access (Windows Security log)
System logOS events: service start/stop, driver failures, reboots
Application logEvents written by individual applications
Linux logs/var/log/syslog or /var/log/messages (general), /var/log/auth.log or /var/log/secure (authentication)
Firewall logsAllowed/blocked connections — source/destination IP, port, protocol
Packet capturesFull traffic content via protocol analyzer (Wireshark, tcpdump); includes payloads + metadata (headers: who, when, where)

Syslog & SIEM

  • Syslog UDP 514 — standard protocol for shipping log entries to a central collector. Format: facility, severity (0 = Emergency → 7 = Debug), message.
  • SIEM (Security Information and Event Management) — aggregates logs from many sources, correlates events across them, normalizes time (NTP-synced), alerts in near-real-time, and presents dashboards. Long-term archival supports investigations.
  • A SIEM is detective, not preventive — it surfaces what happened; an analyst or automation responds.
Exam cue: "correlate events from multiple servers/devices in one place" = SIEM. "Capture and inspect packet contents" = protocol analyzer / packet capture.

Chapter 2 — Identity & Access Management

Who are you, and what can you touch

The AAA Framework (+ Identification)

Four distinct steps — keep them separate
StepQuestion AnsweredExample
Identification"Who do you claim to be?"Typing a username
Authentication"Prove it."Password + TOTP code
Authorization"What are you allowed to do?"ACL grants read on the share
Accounting"What did you actually do?"Authentication logs, audit trail of access
Trap: a username alone never authenticates — it only identifies. Authorization happens after successful authentication, never before.

Authentication Factors & MFA

Different factors, not two of the same kind
Factor"Something you…"Examples
KnowledgeknowPassword, PIN, KBA security questions (weakest factor — answers are researchable)
PossessionhaveSmart card, security key (FIDO2/passkey), hard token, soft token / authenticator app
InherenceareFingerprint, face, iris, vein, gait — biometrics
Locationare somewhereGeolocation/geofencing as an additional signal
MFA rule: two factors must come from different categories. Password + PIN = still single-factor (both knowledge). Password + authenticator code = true 2FA (know + have).

Password policy mechanics

  • Password expiration — forces periodic change; modern guidance (NIST 800-63B) de-emphasizes routine expiration in favor of length + breach screening.
  • Password history — remembers previous N passwords so users can't recycle them (pair with minimum age, or users cycle through instantly).
  • Account lockout policy — threshold (e.g., 5 wrong attempts) triggers lock for a duration (e.g., 30 min). Defeats online brute-force; doesn't help against offline cracking of stolen hashes.
  • Password manager / vault — enables long unique passwords per site; the practical fix for password reuse.

HOTP vs TOTP

One letter, one key difference

HOTP — HMAC-based One-Time Password

Counter-based. Code stays valid until used — no expiration. Counter increments on each use.

Risk: an unused code can be phished and replayed later.

TOTP — Time-based One-Time Password

Time-based. Code expires automatically, typically every 30 seconds. What Google Authenticator / Microsoft Authenticator use.

Requires reasonably synced clocks.

Exam cue: "the code expired before the user typed it" = TOTP. "code remains valid until used" = HOTP. Hard token = dedicated device; soft token = app on your phone.

Biometric Error Rates

FAR vs FRR vs CER — guaranteed exam fodder
MetricMeaningSecurity Impact
FAR — False Acceptance RateSystem accepts an impostorSecurity failure — unauthorized person gets in
FRR — False Rejection RateSystem rejects a legitimate userUsability failure — locked-out employees, helpdesk load
True acceptance / true rejectionCorrect accept of legit user / correct reject of impostorThe desired outcomes
CER — Crossover Error RateThe point where FAR = FRRLower CER = more accurate system. Use CER to compare biometric products.
Tuning trade-off: increase sensitivity → FRR rises, FAR falls (more secure, more annoying). Decrease sensitivity → FAR rises (less secure). The exam loves asking which direction to tune for a "high-security facility" (accept higher FRR).

Account Lifecycle

Provision → audit → deprovision
  • Identity proofing — verifying a person is who they claim before issuing credentials (e.g., ID documents at onboarding, verified questions before a password reset).
  • Provisioning — creating the account and granting the access the role needs (and nothing more).
  • Account types — user, privileged/admin (separate from daily-driver account), service accounts (no interactive login), guest, shared (avoid — destroys accountability).
  • Account audits / attestation — periodic review where a manager formally confirms each user still needs their access. Primary tool for catching privilege creep.
  • Deprovisioning / offboarding — disable the account at termination (don't delete immediately — preserves encrypted data access and audit trail), revoke tokens/keys, recover assets.
  • Time-based logins — restrict login hours (e.g., no 3 a.m. logins for day-shift staff). Concurrent session limits — flag the same account logged in from two places.
Terminated-employee question: the BEST first step is disable the account (and do it during the exit interview for involuntary terminations) — not delete, not "change the password."

Privileged Access Management & Least Privilege

Admin rights are the crown jewels

PAM — Privileged Access Management

Vaulting, monitoring, and controlling admin credentials. Features: credential check-out, session recording, automatic password rotation.

Just-in-Time (JIT) permissions: admin rights granted only when needed, for a limited window, then automatically revoked. No standing privileges.

Least Privilege & Privilege Creep

Least privilege: grant only the access required to do the job — applies to users, processes, and service accounts.

Privilege creep: access accumulating as people change roles without old rights being removed. Detected by account audits/attestation; prevented by role-based provisioning.

SSO, Federation & the Protocols

Who vouches for you, and how

SSO — authenticate once, access many systems. Fewer passwords, but the account becomes a single point of compromise → protect it with MFA.

Federation — SSO across organizations that trust each other. Three parties:

RoleFunction
PrincipalThe user trying to access a resource
Identity Provider (IdP)Authenticates the user and issues assertions (e.g., your company's Entra ID/Okta)
Service Provider (SP)The app/resource that trusts the IdP's assertion instead of doing its own login
ProtocolPurposeKey Detail
SAMLAuthentication (and authorization data) for web SSOXML-based assertions between IdP and SP
OAuth 2.0Authorization — delegated access"Let app A read my data in app B" without sharing the password; issues access tokens
LDAP 389 / LDAPS 636Directory protocol — query/modify directory services (Active Directory)X.500-style distinguished names: CN=Jared,OU=Students,DC=sjsu,DC=edu
Memory hook: SAML = authentication ("who you are"), OAuth = authorization ("what an app may do on your behalf"). "Log in with Google" on a third-party site = federation in action.

ModelAccess Decided ByTelltale Exam Phrase

IndicatorWhat It Looks Like / Means
Impossible travelLogin from New York at 9:00, from Moscow at 9:30 — physically impossible → credential compromise. Detected by geolocation correlation.
Account lockout spikesMany failed attempts across accounts → password spraying or brute force.
